To transform your Secure Quiz from a specific class tool into a **Universal SaaS (Software as a Service) platform** (like Kahoot, Quizizz, or Google Forms), you need to shift from a "hardcoded" architecture to a "dynamic" architecture. 

Currently, a teacher has to edit `script.js` to add students and change the `WEB_APP_URL`, and upload a specific `questions.json` file. 

Here is the step-by-step roadmap to make it a universal platform where **any teacher** can sign up, create a quiz, and share a link with their students.

---

### Step 1: Create a "Teacher Dashboard" (The Missing Piece)
Right now, you only have the **Student View** (`index.html`). You need to build a separate interface (e.g., `teacher.html` or a separate React app) where educators can manage their quizzes.

The Teacher Dashboard should allow them to:
1. **Create an Account:** (Using Firebase Auth or Supabase).
2. **Create a Quiz:** A form where they can type questions, options, and correct answers, OR upload a JSON/CSV file.
3. **Set Rules:** Toggles for Time Limit, Max Tab Switches, and Strict Mode.
4. **Manage Roster:** Upload a list of allowed Student IDs/Emails (replacing your hardcoded `STUDENT_REGISTRY`).
5. **Generate a Link:** E.g., `yourwebsite.com/quiz?id=A8X92F`.
6. **View Results:** A table showing student scores and proctoring violations (tab switches, etc.).

---

### Step 2: Use URL Parameters for Dynamic Routing
Instead of hardcoding the quiz data, your `index.html` (the student view) needs to know *which* quiz to load based on the URL.

If a student visits `securequiz.com/?quizId=MATH101`:
```javascript
// Inside script.js
const urlParams = new URLSearchParams(window.location.search);
const quizId = urlParams.get('quizId'); // Gets "MATH101"

if (!quizId) {
    showError("Invalid Quiz Link. Please check your URL.");
} else {
    // Fetch quiz data from backend using the ID
    fetchQuizFromDatabase(quizId); 
}
```

---

### Step 3: Upgrade the Backend (Move away from Google Sheets)
Google Apps Script is amazing for small projects, but it will **fail** if 500 different teachers and 10,000 students try to use it simultaneously (due to rate limits and execution times).

You need a real NoSQL or SQL database. **Firebase (Firestore)** or **Supabase** are the best choices because they are free to start, integrate perfectly with Vanilla JS, and handle real-time data easily.

**New Database Structure:**
* **`users` (collection):** Stores Teacher profiles.
* **`quizzes` (collection):** Stores the quiz settings, questions, time limits, and the roster of allowed students.
* **`submissions` (collection):** Stores the students' answers, scores, and proctoring logs.

---

### Step 4: Externalize the Student Registry
Remove the `const STUDENT_REGISTRY = {...}` from `script.js`. It makes your app impossible to scale and exposes student names to the public.

**New Flow:**
1. Student types their USN and hits "Start".
2. Frontend sends the USN and `quizId` to the backend.
3. Backend checks the `quizzes` database.
4. If the USN is in that specific quiz's allowed roster, the backend returns a success token and the quiz starts. Otherwise, it throws an error ("You are not registered for this quiz").

---

### Step 5: Secure the Scoring (Backend Scoring)
Currently, `questions.json` contains the correct answers. Any student taking the quiz can open the Network tab, download the JSON, and see the answers before starting.

**To make it universally secure:**
1. When the student loads the quiz, the backend sends the questions and options, but **strips out the `answer` key**.
2. When the timer ends, the frontend sends an array of the student's choices: `[2, 0, 1, 3]`.
3. **The Backend** compares those choices to the hidden answer key, calculates the score, and saves it to the database.
4. The Backend sends the final score back to the frontend to display on the Result Screen.
